二、持久化

RDB vs AOF

方面RDBAOF
原理定时快照追加日志
数据安全可能丢几分钟最多丢1秒
文件大小小（压缩）大
恢复速度快慢
推荐：
- 重要数据：两者都开
- 允许丢几分钟：只用 RDB
- 不允许丢数据：只用 AOF

三、分布式锁

基本实现

加锁：
SET lock_key unique_value NX PX 30000

参数：
- NX：不存在才设置（互斥）
- PX：过期时间（防止死锁）
- unique_value：唯一标识（防止误删）

解锁（Lua 脚本）：
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end

Redisson 特性

- 看门狗机制：自动续期
- 可重入锁：同一线程可多次加锁
- 公平锁：按请求顺序获取
- 读写锁：读写分离

四、缓存问题

缓存穿透

问题：查询不存在的 key，请求打到数据库

解决：
1. 缓存空值：结果为空也缓存，设短过期时间
2. 布隆过滤器：预热所有存在的 key
3. 参数校验：过滤不合法请求

缓存击穿

问题：热点 key 过期瞬间，大量请求打到数据库

解决：
1. 热点数据永不过期
2. 加互斥锁：只有一个请求查数据库

缓存雪崩

问题：大量 key 同时过期，或 Redis 宕机

解决：
1. 过期时间加随机值
2. 多级缓存：Redis + 本地缓存
3. 高可用部署：哨兵或集群

五、实战说辞

Redis 使用场景

我们项目用 Redis 主要有几个场景：

  1. 缓存：用户信息、配置信息缓存，减轻数据库压力
  2. 分布式锁：购物车结算、支付回调用分布式锁保证幂等性
  3. 购物车：用 Hash 结构存购物车数据，field 是商品 ID
  4. 消息队列：学情统计任务用 List 结构做队列

分布式锁实现

我们购物车结算用 Redis 分布式锁。

加锁用 SET lock_key unique_value NX PX 30000，NX 保证不存在才设置，PX 设置过期时间防止死锁。

解锁用 Lua 脚本：先检查 value 是否是自己设置的，再删除。这样防止误删别人的锁。

我们没用 Redisson，是自己实现的。主要考虑业务场景相对简单，自研够用。

六、可能追问

Q: Redis 为什么快？

1. 单线程：不用线程切换开销，不用锁竞争
2. IO 多路复用：epoll 机制，单线程处理大量连接
3. 内存操作：比磁盘快很多
4. 高效数据结构：SDS、ziplist、skiplist 等

Q: 缓存与数据库一致性怎么处理？

推荐方案：先更新数据库，再删除缓存

问题：删除缓存失败怎么办？
解决1：重试机制，失败时发消息到 MQ 异步删除
解决2：订阅 Binlog，用 Canal 发现数据更新后删除缓存

延时双删：
1. 先删除缓存
2. 更新数据库
3. 延时 N 毫秒再删除缓存
