二、高频技术考点备战

【Java 基础】★★★★★ 必问

HashMap 源码阅读经历

我主要看过 HashMap、ConcurrentHashMap、ThreadPoolExecutor 这几个核心源码。

HashMap 我关注了它的扩容机制，特别是 1.8 的优化。以前扩容要重新计算每个元素的 hash 值，现在直接用 hash & oldCap 判断，要么位置不变，要么新位置是原索引加旧容量。这个优化减少了 hash 计算开销。

另外 1.7 的头插法在并发扩容时可能死循环，1.8 改成尾插法解决这个问题。这个让我理解了为什么不能在多线程环境下用 HashMap。

ConcurrentHashMap 我看过 1.8 的实现，用 CAS + synchronized 替代了 1.7 的分段锁。锁粒度更细，只锁链表头节点，并发度更高。这个设计思路很值得学习。

【JVM】★★★★★ 必问

JVM 调优经验

有过线上排查问题的经验。我们有个服务高峰期响应变慢，我通过 jstat 发现 Full GC 比较频繁。

进一步用 jmap -histo 分析对象分布，发现有个报表导出功能创建了大量大对象。这些对象直接进了老年代，导致老年代快速填满，触发 Full GC。

解决方案是把导出改成流式处理，分批查询写入，不再一次性加载所有数据到内存。改完后 Full GC 频率明显下降。

【Spring】★★★★★ 必问

Spring Bean 生命周期

我了解 Spring Bean 的完整生命周期，大致分为实例化、属性赋值、初始化、使用、销毁这几个阶段。

实例化阶段创建 Bean 对象，然后进行依赖注入。初始化阶段会依次调用 Aware 接口、BeanPostProcessor、初始化方法（@PostConstruct）。BeanPostProcessor 的后置处理阶段可能生成 AOP 代理。销毁阶段调用 @PreDestroy 等方法。

【MySQL】★★★★★ 必问

慢 SQL 优化案例

有次有个学情统计查询很慢，执行要好几秒。

我用 EXPLAIN 分析，发现 type 是 ALL，全表扫描。原因是 WHERE 条件用了一个日期格式化函数 YEAR(create_time)，导致索引失效。

我改了 SQL，把日期计算移到代码里，改成 WHERE create_time >= '2023-01-01' AND create_time < '2024-01-01'，这样就走了日期索引。改完后查询时间降到了几百毫秒。

【Redis】★★★★★ 必问

分布式锁实现

我们购物车结算用 Redis 分布式锁。

加锁用 SET lock_key unique_value NX PX 30000，NX 保证不存在才设置，PX 设置过期时间防止死锁。

解锁用 Lua 脚本：先检查 value 是否是自己设置的，再删除。这样防止误删别人的锁。

我们没用 Redisson，是自己实现的。主要考虑业务场景相对简单，自研够用。如果要支持自动续期、可重入，用 Redisson 更方便。

【Netty】★★★ 亮点（区分度高）

Netty 使用场景

我们项目用 Netty 做考勤设备的长连接通信。

设备通过 TCP 连接到服务器，上报考勤数据。用 Netty 的原因是：设备数量多，需要高并发处理；长连接需要心跳保活；原生 NIO 编程太复杂。

我们配置了 Boss Group（1个线程）和 Worker Group（4个线程）。Boss 负责接收设备连接，Worker 负责处理设备上报数据。还加了 IdleStateHandler 做心跳检测，30秒没收到数据就断开连接。

【WebFlux】★★★ 亮点

为什么用 WebFlux？

我们用 WebFlux 做实时讲解服务，主要有几个原因：

  1. 需要实时推送（SSE）：学生平板端实时收到讲解内容，WebFlux 天然支持 SSE。
  2. 调用外部 AI 接口：AI 接口响应慢（几秒），传统 Servlet 会阻塞线程等待。WebFlux 异步非阻塞，线程不用等，可以处理其他请求。
  3. 需要合并多个数据流：讲解内容来自 AI（SSE），音频来自 TTS 服务，需要合并后推送。Flux.merge 很方便实现。

但我们只在这个特定场景用 WebFlux，其他业务模块还是用 Spring MVC。因为 WebFlux 编程模型复杂，调试难，团队学习成本高，不是所有场景都需要。
