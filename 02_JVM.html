02. JVM

一、JVM 内存结构

1. 内存区域划分

线程共享区域：

  • 方法区（Metaspace）：类信息、常量、静态变量
  • 堆（Heap）：对象实例、数组，GC 主要区域

线程私有区域：

  • 程序计数器：当前执行的字节码指令地址
  • 虚拟机栈：方法调用的栈帧
  • 本地方法栈：Native 方法调用

2. 堆内存结构

堆 = 新生代 + 老年代

新生代：
- Eden 区（80%）
- Survivor 0 / From 区（10%）
- Survivor 1 / To 区（10%）

老年代：
- 存放长期存活的对象
- 默认占堆的 2/3

对象晋升规则：
- Eden → Survivor：Eden 满时，存活对象复制到 Survivor
- Survivor → 老年代：年龄达到阈值（默认 15），晋升到老年代

二、垃圾回收机制

1. 判断对象可回收

可达性分析法：

从 GC Roots 开始向下搜索
不在引用链上的对象可回收

GC Roots 包括：
1. 虚拟机栈中的引用对象（方法中的局部变量）
2. 方法区中静态属性引用的对象
3. 方法区中常量引用的对象
4. 本地方法栈中 JNI 引用的对象
5. 同步锁持有的对象

2. 四种引用类型

类型特点场景
强引用永不回收普通对象
软引用内存不足时回收缓存
弱引用下次 GC 时回收ThreadLocal
虚引用跟踪对象回收ReferenceQueue

3. 常见垃圾收集器

Serial 收集器

特点：
- 单线程收集
- 收集时暂停所有工作线程（STW）
- 新生代用复制算法，老年代用标记-整理

适用：单核 CPU 或小内存

CMS 收集器

特点：
- 关注低停顿时间
- 老年代收集器，基于标记-清除
- 并发收集（GC 线程和用户线程同时运行）

流程：
1. 初始标记（STW）
2. 并发标记
3. 重新标记（STW）
4. 并发清除

问题：
- 内存碎片
- 浮动垃圾
- Concurrent Mode Failure

G1 收集器

特点：
- JDK 9 默认收集器
- 区域化收集（Region）
- 可预测停顿时间

优点：
- 无碎片（Region 内用复制算法）
- 可预测停顿
- 大对象处理优化

三、JVM 调优

1. 常用 JVM 参数

内存参数：
-Xms：初始堆大小
-Xmx：最大堆大小
-Xmn：新生代大小
-Xss：线程栈大小（默认 1M）

GC 参数：
-XX:+UseG1GC：G1 收集器
-XX:MaxGCPauseMillis=N：最大 GC 停顿时间

2. 常用诊断工具

jstat -gcutil  1000 10  # 查看 GC 统计
jmap -histo            # 对象统计
jmap -dump:file=heap.hprof  # 导出堆转储
jstack                 # 线程栈

四、实战说辞

JVM 调优经验

有过线上排查问题的经验。我们有个服务高峰期响应变慢，我通过 jstat 发现 Full GC 比较频繁。

进一步用 jmap -histo 分析对象分布，发现有个报表导出功能创建了大量大对象。这些对象直接进了老年代，导致老年代快速填满，触发 Full GC。

解决方案是把导出改成流式处理，分批查询写入，不再一次性加载所有数据到内存。改完后 Full GC 频率明显下降。

五、可能追问

Q: 对象什么时候进老年代？

1. 年龄达到阈值：在 Survivor 区经历一定次数 GC（默认 15），晋升到老年代
2. 大对象直接进老年代：超过 -XX:PretenureSizeThreshold 设置的大小
3. Survivor 空间不足：Survivor 区放不下 Eden 区存活对象时，直接晋升

Q: 什么是内存泄漏？怎么排查？

内存泄漏：对象无法被回收，持续占用内存，最终可能 OOM。

排查步骤：
1. 导出堆转储：jmap -dump:file=heap.hprof
2. 用 MAT 分析，看哪些对象占用最多
3. 查看 GC Roots 引用链，找出是谁引用了这些对象
4. 定位代码，找出为什么这些对象没有被释放
