面试说辞配套文档

面试说辞配套文档

本文档针对简历中各技术点的面试问题提供回答思路，帮助你应对面试追问。

一、Java 基础相关

Q: 你读过哪些源码？有什么收获？

说辞：

我主要看过 HashMap、ConcurrentHashMap、ThreadPoolExecutor 这几个核心类的源码。

HashMap，我重点关注了它的扩容机制和链表转红黑树的阈值。在 1.7 版本存在死循环问题，1.8 改进了这个问题。这个让我理解了为什么在多线程环境下要用 ConcurrentHashMap。

ConcurrentHashMap 我看过它的分段锁设计（1.7）和 CAS+synchronized（1.8）的实现差异，这让我对并发编程有了更深的理解。

ThreadPoolExecutor 我关注了线程池的参数含义、任务提交流程、拒绝策略等。这在实际项目中选择线程池参数时很有帮助。

Q: 多线程项目中遇到过什么问题？

说辞：

在学情统计模块，我们用线程池来并行计算多个班级的数据。遇到过一个问题：某个班级数据量特别大，导致一个线程执行时间很长，其他线程都完成了就等它一个。

后来我们做了优化，把大任务拆分成更细粒度的子任务，用 CompletableFuture 来编排，这样整体效率提升了不少。同时我们监控线程池的队列长度，避免任务堆积。

二、JVM 调优相关

Q: 你有 JVM 调优的经验吗？

说辞：

有过实际排查线上问题的经验。我们有个服务在高峰期偶尔出现响应慢的情况，我通过以下步骤排查：

  1. 先用 jstat -gcutil 查看 GC 情况，发现 Full GC 比较频繁
  2. 用 jmap -histo 查看对象分布，发现某个大对象占用内存较多
  3. 结合代码分析，发现是一个报表导出功能一次性加载了太多数据到内存

最后的解决方案是改成流式处理，分批查询、分批写入，问题就缓解了。虽然没有精确统计优化比例，但从监控上看 Full GC 频率明显下降了。

Q: 你们项目的 JVM 参数是怎么配置的？

说辞：

我们一般是基于默认配置做调整。主要关注几个参数：

  • Xms 和 Xmx 设置成一样，避免内存抖动
  • 新生代比例根据对象生命周期特点调整，短生命周期对象多的服务会适当调大新生代
  • GC 收集器我们用 G1，因为它的停顿时间相对可控

三、Redis 相关

Q: Redis 分布式锁是怎么实现的？

说辞：

我们项目里用 Redis 实现分布式锁，主要场景是购物车结算和订单支付。

实现方式是用 SET key value NX PX timeout 这个原子命令，NX 保证不存在才设置，PX 设置过期时间防止死锁。

释放锁时用 Lua 脚本保证原子性，先判断 value 是不是自己设置的，再执行 DEL，防止误删别人的锁。

Q: Redis 用在哪些场景？

说辞：

  • 缓存：热点数据缓存，比如用户信息、配置信息，减轻数据库压力
  • 分布式锁：购物车、订单等场景保证幂等性
  • 消息队列：用 List 结构做简单的异步任务队列，比如学情统计任务
  • 计数器：PV/UV 统计，用 HyperLogLog

Q: Redis 遇到过什么问题？

说辞：

遇到过缓存穿透的问题。有次发现大量请求直接打到数据库，查了日志发现是请求了不存在的 key。

我们的解决方案是：对于查询结果为空的 key 也缓存一个 null 值（设置较短过期时间），同时在入口层做参数校验，过滤掉明显不合法的请求。

四、MySQL 优化相关

Q: 慢 SQL 优化有具体案例吗？

说辞：

有。我们有个学情统计的查询，关联了好几张表，一开始要查好几秒。

我用 EXPLAIN 分析后发现：

  1. 有个字段没建索引，导致全表扫描
  2. 有个关联查询的 ON 条件用到了函数，索引失效了

优化方案：

  1. 给常用的查询条件加联合索引
  2. 把函数计算移到应用层，数据库只做简单查询
  3. 对于复杂的统计查询，改成先用子查询过滤数据，再关联

优化后查询时间从几秒降到了几百毫秒。

五、Elasticsearch 相关

Q: 为什么要用 ES？数据库模糊查询不行吗？

说辞：

数据库的 LIKE 查询有几个问题：

  1. 走不了索引，大数据量下性能很差
  2. 不支持复杂的分词搜索，用户体验不好

我们用 ES 主要解决题目搜索、知识点搜索的需求。配合 IK 分词器，可以支持中文分词，搜索结果更准确。

数据同步方案是：数据变更时发消息到 MQ，消费者负责同步到 ES，保证最终一致性。

六、Netty 相关

Q: Netty 用在什么场景？

说辞：

我们用 Netty 做考勤设备的长连接通信。设备通过 TCP 连接到服务器，上报考勤数据。

用 Netty 的原因是：

  1. 原生 NIO 编程太复杂，Netty 封装得更好
  2. 自带各种编解码器，处理协议方便
  3. 性能好，适合大量长连接场景

Q: Netty 有什么需要注意的？

说辞：

  • 内存泄漏：ByteBuf 如果不释放会内存泄漏，我们用 SimpleChannelInboundHandler，它会自动释放
  • 心跳机制：长连接需要心跳保活，我们设置了读空闲和写空闲超时
  • 断线重连：客户端需要实现断线重连逻辑

七、WebFlux 相关

Q: 为什么用 WebFlux？和传统 Spring MVC 有什么区别？

说辞：

我们用 WebFlux 做实时讲解服务，主要是考虑到：

  1. 需要 SSE 推送，WebFlux 天然支持
  2. 调用第三方 AI 接口时有等待时间，用响应式可以非阻塞等待，提高并发能力

区别：

  • Spring MVC 是同步阻塞模型，一个请求占用一个线程
  • WebFlux 是异步非阻塞，少量线程就能处理大量连接

但 WebFlux 编程模型比较难调试，我们只在对性能要求高的场景用，其他模块还是用传统的 Spring MVC。

八、DDD 相关

Q: 你们项目是怎么做 DDD 的？

说辞：

我们不是严格的 DDD，更多是借鉴 DDD 的思想来做业务建模。

主要做法：

  1. 领域划分：比如知识树、组题引擎、错题册，每个领域有自己的实体和业务规则
  2. 聚合根设计：比如"作业"是聚合根，包含题目、答案、批阅结果等
  3. 领域服务：跨聚合的业务逻辑放在领域服务里
  4. 防腐层：对接第三方系统时，通过防腐层隔离，不让外部模型污染领域模型

好处是业务逻辑清晰，代码结构更好维护。但代价是前期设计成本高，需要和产品反复对齐业务概念。

九、分布式相关

Q: 你们高并发场景是怎么处理的？

说辞：

我们的项目有明显的时段高峰，比如放学后学生集中做作业。

应对方案：

  1. 缓存：热点数据缓存到 Redis
  2. 异步处理：非核心流程异步化，比如学情统计、日志记录
  3. 限流：接口层做限流，保护后端服务
  4. 降级：非核心功能降级，比如个性化推荐暂时关闭
  5. 队列削峰：高峰期请求先进队列，后端慢慢消费

Q: 分布式锁保证幂等性，能具体说说吗？

说辞：

场景：支付成功后发放权益，不能重复发放。

方案：

  1. 支付回调时，用订单号作为锁的 key
  2. 获取到锁之后，先查数据库看订单状态是否已处理
  3. 未处理则执行业务逻辑，更新状态
  4. 释放锁

双重保障：分布式锁防并发，数据库乐观锁（version 字段）做兜底。

十、注意事项

  1. 不要背答案：以上说辞是思路，面试时要用自己的话表达
  2. 诚实为上：如果问到不会的，可以说"这个我了解但没深入用过"，然后说说自己的理解
  3. 举一反三：面试官可能会追问细节，要能灵活应对
  4. 准备具体场景：每个技术点都要有"在什么场景、遇到什么问题、怎么解决"的故事
