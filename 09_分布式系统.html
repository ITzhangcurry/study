09. 分布式系统

一、分布式基础

1. CAP 理论

CAP 理论：分布式系统最多同时满足三项中的两项

C（Consistency）一致性：
- 所有节点数据一致
- 读操作返回最新写入的值

A（Availability）可用性：
- 每个请求都能得到响应（成功或失败）
- 不会超时或拒绝

P（Partition Tolerance）分区容错性：
- 网络分区（节点间通信中断）时系统仍能运行

为什么最多满足两项？
- 发生分区（P）时，要么放弃一致性（C），要么放弃可用性（A）
- 因为分区时，部分节点无法通信，数据可能不一致
- 要一致就要等待网络恢复（不可用）
- 要可用就要接受数据不一致

选择：
- CP：放弃可用性，保证一致性（如 Zookeeper、HBase）
- AP：放弃一致性，保证可用性（如 Cassandra、DynamoDB）
- CA：不可能（单机系统）

2. BASE 理论

BASE：Basically Available, Soft state, Eventually consistent

基本可用（Basically Available）：
- 系统出现故障时，允许损失部分可用性
- 如响应时间变长、部分功能降级

软状态（Soft state）：
- 允许数据存在中间状态
- 不要求实时一致

最终一致性（Eventually consistent）：
- 经过一段时间后，数据最终一致
- 不保证实时一致，但保证最终一致

与 ACID 对比：
- ACID：强一致性，传统数据库
- BASE：最终一致性，分布式系统

二、分布式锁

1. Redis 分布式锁

基本实现：
SET lock_key unique_value NX PX 30000

参数：
- NX：不存在才设置（互斥）
- PX 30000：过期时间 30 秒（防止死锁）
- unique_value：唯一标识（防止误删）

解锁（Lua 脚本）：
if redis.call("get", KEYS[1]) == ARGV[1] then
    return redis.call("del", KEYS[1])
else
    return 0
end

问题：
1. 锁过期但业务没执行完
   解决：Redisson 看门狗自动续期

2. 主从切换导致锁丢失
   解决：RedLock（多 Redis 实例）

3. 客户端宕机
   解决：设置过期时间，自动释放

2. Zookeeper 分布式锁

Zookeeper 分布式锁原理：

方案：临时顺序节点
- 创建临时顺序节点 /lock/node-xxx
- 序号最小的节点获得锁
- 监听前一个节点的删除事件
- 优点：避免羊群效应，顺序获取锁

流程：
1. 创建临时顺序节点
2. 判断自己序号是否最小
   - 最小：加锁成功
   - 不是最小：监听前一个节点
3. 前一个节点删除，重新判断
4. 业务完成，删除自己的节点

优点：
- 可靠（Zookeeper 保证一致性）
- 有序（顺序节点）
- 无死锁（临时节点，客户端断开自动删除）

缺点：
- 性能不如 Redis
- 维护复杂（需要 Zookeeper 集群）

三、分布式事务

1. 两阶段提交（2PC）

两阶段提交（2 Phase Commit）：

角色：
- 协调者（Coordinator）：管理事务
- 参与者（Participant）：执行事务

流程：
第一阶段（Prepare）：
1. 协调者向所有参与者发送 Prepare 请求
2. 参与者执行事务，但不提交
3. 参与者将结果返回协调者（Yes/No）

第二阶段（Commit/Abort）：
4. 如果所有参与者都返回 Yes：
   - 协调者发送 Commit
   - 参与者提交事务
5. 如果有参与者返回 No：
   - 协调者发送 Abort
   - 参与者回滚事务

问题：
1. 同步阻塞：参与者等待协调者期间阻塞
2. 单点故障：协调者宕机，参与者一直阻塞
3. 数据不一致：协调者发送 Commit 后宕机，部分参与者没收到

2. TCC（Try-Confirm-Cancel）

TCC：业务层面的分布式事务

三个阶段：
- Try：预留资源（冻结金额、锁定库存）
- Confirm：确认执行（扣款、扣库存）
- Cancel：取消执行（解冻、释放库存）

流程：
1. Try：所有参与者预留资源
2. 如果全部成功：
   - Confirm：确认执行
3. 如果部分失败：
   - Cancel：取消执行

要求：
- Try 成功，Confirm 必须成功
- Try 失败，Cancel 必须成功
- Confirm/Cancel 要幂等（可能重试）

优点：
- 锁粒度小（只锁需要的资源）
- 性能好

缺点：
- 业务侵入（要写 Try/Confirm/Cancel）
- 实现复杂

3. 本地消息表

本地消息表：最终一致性方案

流程：
1. 业务操作 + 写入本地消息表（同一事务）
2. 定时任务扫描本地消息表，发送消息到 MQ
3. 参与者消费消息，执行操作
4. 参与者执行成功，返回 ACK
5. 本地消息表状态更新为已处理

优点：
- 简单
- 不依赖复杂的事务框架
- 最终一致性

缺点：
- 需要维护本地消息表
- 有延迟（定时任务扫描）

四、幂等性设计

幂等方案

幂等：多次执行同一操作，结果相同

方案一：数据库唯一索引
CREATE TABLE orders (
    id VARCHAR(64) PRIMARY KEY
);
流程：
1. 请求携带订单号
2. 尝试插入
3. 重复插入失败（唯一约束冲突）

方案二：分布式锁 + 状态检查
流程：
1. 加分布式锁（订单号为 key）
2. 查询订单状态
3. 未处理：执行业务，更新状态为已处理
4. 已处理：返回结果
5. 释放锁

方案三：Token 机制
流程：
1. 客户端先申请 token
2. 请求携带 token
3. 服务端验证 token，删除 token
4. 重复请求（token 已删除）返回错误

方案四：乐观锁
UPDATE orders
SET status = 'PAID', version = version + 1
WHERE id = 'xxx' AND version = 1;

五、分布式 ID

Snowflake（雪花算法）

Snowflake：64 位 ID

结构：
| 1 位 | 41 位时间戳 | 10 位机器 ID | 12 位序列号 |
| 符号位（0）| 毫秒级时间 | 数据中心+机器 | 同一毫秒内序号 |

- 时间戳（41 位）：约 69 年
- 机器 ID（10 位）：1024 台机器
- 序列号（12 位）：同一毫秒 4096 个 ID

优点：
- 有序（时间递增）
- 高性能
- 无单点

缺点：
- 时钟回拨问题（时间倒退）
- 机器 ID 需分配

时钟回拨解决：
- 记录上次时间戳
- 当前时间 < 上次时间：拒绝生成 或 等待

六、实战说辞

分布式锁使用

我们购物车结算用 Redis 分布式锁。

场景：支付回调后发放权益，不能重复发放。

实现：锁的 key 是订单号，加锁用 SET key value NX PX 30000，获取锁后查订单状态，未处理则执行。解锁用 Lua 脚本检查 value，防止误删。

没用 Redisson，因为场景简单，自研够用。另外数据库有乐观锁兜底（version 字段），即使分布式锁失效也不会重复。

分布式事务处理

我们项目大部分场景用最终一致性，不是强一致性。

比如：订单支付后发权益。

方案：订单服务本地事务完成（订单状态改为已支付），发消息到 MQ："订单已支付"。权益服务消费消息，发放权益。失败时 MQ 重试，重试多次失败报警人工处理。

这样订单服务不会被权益服务阻塞，性能更好。

七、可能追问

Q: 分布式锁用 Redis 还是 Zookeeper？

Redis：性能高，简单，但有主从切换丢锁风险
Zookeeper：可靠性高（CP），但性能低，维护复杂

我们选 Redis：
- 项目已有 Redis，不用额外维护
- 场景简单，Redis 性能够用
- 有数据库乐观锁兜底

Q: 如何处理消息消费幂等？

方案：
1. 业务表唯一索引：消费时插入唯一记录，重复插入失败
2. 状态表：消费前检查是否已消费，记录消费状态
3. 分布式锁：消费时加锁，锁内检查状态

我们用的是状态表方案：消息有唯一 message_id，消费时先查是否存在，不存在则执行并插入。
