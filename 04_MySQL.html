04. MySQL

一、索引详解

1. B+ 树结构

B+ 树特点：
1. 非叶子节点只存键值和指针，不存数据
   → 一层能存更多节点，树更矮，IO 更少

2. 叶子节点存数据，并用指针连接成链表
   → 范围查询效率高

3. 所有查询都要到叶子节点
   → 查询效率稳定

为什么 MySQL 用 B+ 树：
- 范围查询只需遍历叶子节点
- 树更矮，减少 IO 次数

2. 聚簇索引 vs 辅助索引

聚簇索引（主键索引）：
- 叶子节点存完整行数据
- 一张表只有一个聚簇索引

辅助索引（非主键索引）：
- 叶子节点存主键值
- 查询流程：辅助索引 → 主键值 → 聚簇索引（回表）

覆盖索引：
- 查询列都在辅助索引中
- 不需要回表

3. 最左匹配原则

组合索引：(a, b, c)

可以走索引：
- WHERE a = 1
- WHERE a = 1 AND b = 2
- WHERE a = 1 AND b = 2 AND c = 3

不能走索引：
- WHERE b = 2（缺少最左列）
- WHERE a = 1 AND c = 3（b 缺失，c 用不了）

原因：
- 索引按 (a, b, c) 顺序排列
- 先按 a 排序，再在 a 相同的组里按 b 排序

4. 索引失效场景

1. 使用函数：WHERE YEAR(create_time) = 2023
   → 改为：WHERE create_time >= '2023-01-01'

2. 类型转换：WHERE id = '123'（id 是 INT）

3. LIKE 左模糊：WHERE name LIKE '%张'

4. NOT IN：改为 NOT EXISTS

5. 范围查询后的列：(a, b, c)
   WHERE a = 1 AND b > 2 AND c = 3
   → c 不走索引

2. MVCC 原理

核心组件：
1. 隐藏字段：DB_TRX_ID、DB_ROLL_PTR
2. Undo Log：记录历史版本，形成版本链
3. Read View：判断哪些版本可见

可见性判断：
- DB_TRX_ID < min_trx_id：可见
- DB_TRX_ID >= max_trx_id：不可见
- DB_TRX_ID 在 m_ids 中：不可见（未提交）
- DB_TRX_ID 不在 m_ids 中：可见（已提交）

3. 锁类型

行锁（Record Lock）：锁单行记录
间隙锁（Gap Lock）：锁一个范围，不锁记录本身
临键锁（Next-Key Lock）：Record Lock + Gap Lock

RR 默认用 Next-Key Lock，防止幻读
RC 只有 Record Lock，可能幻读

三、SQL 优化

1. EXPLAIN 分析

type：访问类型
- const：单行，主键/唯一索引
- ref：使用普通索引
- range：范围扫描
- index：全索引扫描
- ALL：全表扫描（需要优化）

Extra：
- Using index：覆盖索引（好）
- Using temporary：临时表（需优化）
- Using filesort：文件排序（需优化）

2. 索引设计原则

1. 选择合适的列：WHERE、JOIN、ORDER BY 列
2. 组合索引：最左列最常用，区分度高放左边
3. 使用覆盖索引：查询列都在索引中
4. 单表索引不超过 5 个

四、实战说辞

慢 SQL 优化案例

有次有个学情统计查询很慢，执行要好几秒。

我用 EXPLAIN 分析，发现 type 是 ALL，全表扫描。原因是 WHERE 条件用了一个日期格式化函数 YEAR(create_time)，导致索引失效。

我改了 SQL，把日期计算移到代码里，改成 WHERE create_time >= '2023-01-01' AND create_time < '2024-01-01'，这样就走了日期索引。改完后查询时间降到了几百毫秒。

五、可能追问

Q: 为什么推荐用自增主键？

自增主键的好处：
1. 插入时新数据总是追加到末尾，不会发生页分裂
2. 索引更紧凑，空间利用率高
3. 范围查询效率高

如果用随机值（如 UUID）做主键：
1. 插入位置随机，可能触发页分裂
2. 索引不紧凑，有碎片
3. 页分裂时需要移动数据，写入性能差

Q: 什么是索引下推？

索引下推（ICP）：

没有 ICP：
- 存储引擎返回满足索引条件的记录
- Server 层再根据 WHERE 其他条件过滤

有 ICP：
- 索引条件 + 部分 WHERE 条件都在引擎层过滤
- 只返回满足全部条件的记录

减少回表次数和返回记录数
