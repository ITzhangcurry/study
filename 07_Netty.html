07. Netty

一、Netty 基础

1. 为什么用 Netty？

原生 NIO 的问题：

1. API 复杂
   - Selector、Channel、Buffer 配合使用
   - 需要处理很多细节

2. Epoll Bug
   - Linux 下 Selector 空轮询问题
   - 导致 CPU 100%
   - JDK 未完全修复

3. 拆包粘包
   - TCP 是流式协议，没有消息边界
   - 需要自己处理

4. 断线重连
   - 连接断开需要重新建立
   - 需要自己实现

Netty 解决的问题：

1. API 简洁
   - ChannelHandler 处理 IO 事件
   - 不需要关心底层细节

2. 解决 Epoll Bug
   - 检测到空轮询时重建 Selector

3. 内置编解码器
   - LineBasedFrameDecoder（行分隔）
   - DelimiterBasedFrameDecoder（自定义分隔符）
   - LengthFieldBasedFrameDecoder（长度字段）

4. 高性能
   - 零拷贝
   - 内存池
   - Reactor 线程模型

5. 功能丰富
   - 心跳检测
   - 断线重连
   - SSL/TLS

2. Netty 线程模型

Reactor 模型：

单线程 Reactor：
- 一个线程处理所有 IO
- 适合小负载

多线程 Reactor：
- 一个线程接收连接（Acceptor）
- 多个线程处理 IO（Worker）
- 适合高负载

主从 Reactor（Netty 使用）：
- 主 Reactor：接收连接
- 从 Reactor：处理 IO
- 进一步提高并发

Netty 实现：

Boss Group（主 Reactor）：
- 接收客户端连接
- 通常 1 个线程
- 创建连接后注册到 Worker Group

Worker Group（从 Reactor）：
- 处理 IO 读写
- 多个线程（默认 CPU 核数 * 2）
- 每个线程绑定一个 Selector
- 每个线程处理多个 Channel
- 一个 Channel 只绑定一个线程

流程：
1. Boss 线程接收连接，创建 SocketChannel
2. Boss 线程选择一个 Worker，注册 Channel
3. Worker 线程监听 Channel 的 IO 事件
4. Worker 线程处理读写，触发 ChannelHandler

3. Netty 核心组件

1. Channel
   - 代表一个连接（Socket）
   - 可以读写数据

2. EventLoop
   - 事件循环，处理 IO 事件
   - 一个 EventLoop 对应一个线程
   - 一个 EventLoop 可处理多个 Channel

3. EventLoopGroup
   - EventLoop 的集合
   - Boss Group + Worker Group

4. ChannelPipeline
   - ChannelHandler 的链
   - 处理 IO 事件的流水线

5. ChannelHandler
   - 处理 IO 事件的处理器
   - ChannelInboundHandler：入站（读）
   - ChannelOutboundHandler：出站（写）

6. ByteBuf
   - 字节缓冲区
   - 读写指针分离
   - 支持池化

7. ChannelFuture
   - 异步操作结果
   - 可添加监听器

二、ByteBuf

1. ByteBuf 结构

ByteBuf 三个区域：

+-------------------+------------------+------------------+
|   discard bytes   |  readable bytes  |  writable bytes  |
|     (已读区域)      |    (可读区域)     |    (可写区域)     |
+-------------------+------------------+------------------+

0      <=      readerIndex   <=   writerIndex    <=    capacity

readerIndex：读指针，读取后增加
writerIndex：写指针，写入后增加
capacity：容量

优势（相比 NIO ByteBuffer）：
- 读写指针分离，不需要 flip()
- 可动态扩容
- 支持池化

2. ByteBuf 类型

三种类型：

1. Heap Buffer（堆内存）
   - 在 JVM 堆上分配
   - 有 GC 压力
   - 适合小数据

2. Direct Buffer（直接内存）
   - 在堆外分配
   - 无 GC 压力
   - 适合大数据、网络传输
   - 零拷贝

3. Composite Buffer（复合缓冲区）
   - 多个 ByteBuf 组合
   - 逻辑上是一个 Buffer
   - 避免数据复制

3. 内存泄漏问题

ByteBuf 内存泄漏：

问题：
- Direct Buffer 不受 GC 管理
- 如果不释放，内存泄漏
- Netty 用引用计数管理

引用计数：
- 创建时引用计数 = 1
- retain()：引用计数 +1
- release()：引用计数 -1
- 引用计数 = 0 时释放内存

规则：
- 入站 ByteBuf：SimpleChannelInboundHandler 自动释放
- 出站 ByteBuf：写入 Channel 后自动释放
- 自定义处理：需要手动 release()

三、编解码器

1. 拆包粘包

TCP 拆包粘包：

发送：消息1 + 消息2 + 消息3
可能接收：
- 消息1 + 消息2（粘包）
- 消息1 + 消息2前半 + 消息2后半（拆包）
- 消息1 + 消息2 + 消息3（正常）

解决：
- 固定长度：FixedLengthFrameDecoder
- 分隔符：LineBasedFrameDecoder
- 长度字段：LengthFieldBasedFrameDecoder
- 自定义协议：消息头 + 消息体

2. 常用编解码器

拆包解码器：

1. FixedLengthFrameDecoder
   - 固定长度拆包
   .addLast(new FixedLengthFrameDecoder(100))

2. LineBasedFrameDecoder
   - 换行符拆包（\n 或 \r\n）
   .addLast(new LineBasedFrameDecoder(1024))

3. DelimiterBasedFrameDecoder
   - 自定义分隔符
   ByteBuf delimiter = Unpooled.copiedBuffer("$".getBytes());
   .addLast(new DelimiterBasedFrameDecoder(1024, delimiter))

4. LengthFieldBasedFrameDecoder
   - 长度字段拆包
   // 消息格式：长度(2字节) + 内容
   .addLast(new LengthFieldBasedFrameDecoder(65535, 0, 2, 0, 2))

编解码器：
StringDecoder：ByteBuf → String
StringEncoder：String → ByteBuf

四、高级特性

1. 心跳机制

心跳检测：
- 检测连接是否存活
- 超时则断开

IdleStateHandler：
- 检测读空闲、写空闲、读写空闲
- 触发 IdleStateEvent

使用：
ch.pipeline()
    .addLast(new IdleStateHandler(30, 0, 0, TimeUnit.SECONDS))
    .addLast(new HeartbeatHandler());

2. 断线重连

客户端断线重连：

实现：
- 连接失败时延迟重试
- 连接断开时自动重连
- 用 ChannelFutureListener 监听连接结果

关键代码：
future.addListener(new ChannelFutureListener() {
    @Override
    public void operationComplete(ChannelFuture future) {
        if (future.isSuccess()) {
            channel = future.channel();
        } else {
            // 5秒后重试
            future.channel().eventLoop().schedule(() -> doConnect(), 5, TimeUnit.SECONDS);
        }
    }
});

3. 零拷贝

零拷贝：减少数据复制

Netty 零拷贝方式：

1. Composite ByteBuf
   - 多个 ByteBuf 组合，逻辑上是一个
   - 避免合并时的数据复制

2. slice()
   - ByteBuf 切片，共享底层数据
   - 避免复制

3. FileRegion
   - 文件传输，直接从文件到网络
   - 底层用 OS 的 sendfile

4. Direct Buffer
   - 直接内存，避免 JVM 堆 → 堆外复制

五、实战说辞

Netty 使用场景

我们项目用 Netty 做考勤设备的长连接通信。

设备通过 TCP 连接到服务器，上报考勤数据。用 Netty 的原因是：

  • 设备数量多，需要高并发处理
  • 长连接需要心跳保活
  • 原生 NIO 编程太复杂

我们配置了 Boss Group（1个线程）和 Worker Group（4个线程）。Boss 负责接收设备连接，Worker 负责处理设备上报数据。

还加了 IdleStateHandler 做心跳检测，30秒没收到数据就断开连接。设备端有断线重连逻辑。

编解码处理

我们自定义了消息协议：消息头（4字节长度）+ 消息体（JSON 数据）。

用 LengthFieldBasedFrameDecoder 拆包，先读长度，再读对应长度的内容。

JSON 数据用 StringDecoder 转成 String，再在业务 Handler 里解析成 POJO。

这样处理解决了 TCP 粘包拆包问题。

六、可能追问

Q: Netty 的 Reactor 模型？

Netty 用主从 Reactor 模型：

Boss Group（主 Reactor）：接收连接
- 通常 1 个线程
- 监听 ServerSocketChannel 的 ACCEPT 事件

Worker Group（从 Reactor）：处理 IO
- 多个线程（默认 CPU 核数 * 2）
- 每个 Worker 绑定一个 Selector
- 监听 SocketChannel 的 READ/WRITE 事件

一个 Channel 只绑定一个 Worker，保证同一线程处理，避免并发问题。

Q: Netty 如何解决 Epoll Bug？

JDK NIO 在 Linux 下有 Epoll Bug：Selector 空轮询，CPU 100%。

Netty 的解决方式：
1. 检测空轮询次数
2. 超过阈值（默认 512 次）重建 Selector
3. 把原来的 Channel 注册到新 Selector

这样避免了 CPU 空转问题。

Q: ByteBuf 和 ByteBuffer 的区别？

NIO ByteBuffer：
- 读写指针共用，需要 flip() 切换读写模式
- 不能动态扩容
- 无池化，每次分配新对象

Netty ByteBuf：
- 读写指针分离，readerIndex 和 writerIndex
- 不需要 flip()
- 可动态扩容
- 支持池化（PooledByteBufAllocator）
- 支持零拷贝（Composite、slice）
- 引用计数管理内存
