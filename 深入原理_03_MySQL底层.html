深入原理：MySQL 底层

一、InnoDB 存储结构底层

1. 页结构（Page）

InnoDB 页大小：16KB（默认）

页结构：

| 区段 | 大小 | 说明 |
|------|------|------|
| File Header | 38B | 页通用信息（页号、类型、LSN） |
| Page Header | 56B | 页状态信息（记录数、槽数） |
| Infimum + Supremum | 26B | 最小最大记录（虚拟记录） |
| User Records | 变长 | 用户数据 |
| Free Space | 变长 | 未分配空间 |
| Page Directory | 变长 | 页目录（槽位，加速查找） |
| File Trailer | 8B | 校验和、LSN（校验页完整性） |

记录结构：
| 变长字段长度列表 | NULL值列表 | 记录头信息 | 列值 |

记录头信息（5字节）：
- delete_flag：删除标记
- n_owned：拥有的记录数（页目录）
- next_record：下一条记录偏移量

2. B+ 树节点结构

非叶子节点：
- 页头
- 页目录
- 记录：键值 + 子页号

叶子节点：
- 页头
- 页目录
- 记录：键值 + 数据（聚簇索引）
        或键值 + 主键值（辅助索引）
- next_page：下一叶子页号（范围查询）

页目录：
- 把记录分组，每组最后一条记录偏移量存入页目录
- 每组 4-8 条记录
- 查找时：二分页目录 → 遍历组内记录

3. 行格式

四种行格式：

Compact（紧凑）：
- 变长字段长度列表：存储变长字段实际长度
- NULL 值列表：位图标记 NULL 字段
- 记录头：5 字节
- 列值：固定长度 + 变长长度

Dynamic（动态，默认）：
- 变长字段可能存储在溢出页
- 超过 768 字节的部分存溢出页
- 本页只存前 768 字节 + 溢出页指针

Compressed（压缩）：
- 同 Dynamic，但页压缩

Redundant（冗余）：
- 老格式，不推荐

二、索引底层原理

1. B+ 树 vs B 树

B 树：
- 每个节点都存数据
- 非叶子节点存键值 + 数据 + 子节点指针
- 叶子节点之间无连接

B+ 树：
- 只有叶子节点存数据
- 非叶子节点只存键值 + 子节点指针
- 叶子节点用双向链表连接

B+ 树优势：

1. 单次 IO 获取更多键值
   - 非叶子节点不存数据，一页能存更多键值
   - 树更矮，IO 次数更少
   - 例：16KB 页，键值 8B，指针 6B
   - 一页可存：16KB / (8+6) ≈ 1170 个键值
   - 3 层 B+ 树可存：1170 * 1170 * 16 ≈ 2000 万记录

2. 范围查询高效
   - 叶子节点链表连接
   - 找到起点后，沿链表遍历

3. 查询效率稳定
   - 所有查询都要到叶子节点

2. 聚簇索引 vs 辅助索引

聚簇索引（主键索引）：
- 叶子节点存完整行数据
- 表数据按主键顺序存储
- 一张表只有一个聚簇索引

辅助索引（非主键索引）：
- 叶子节点存主键值
- 查询流程：辅助索引 → 主键值 → 聚簇索引（回表）

回表开销：
- 辅助索引查询需要两次 B+ 树查找
- 例：SELECT * FROM user WHERE name = '张三'
  - name 索引：找到主键值 ID
  - 主键索引：找到完整数据

覆盖索引：
- 查询列都在辅助索引中
- 不需要回表

3. 联合索引底层

联合索引：(a, b, c)

索引结构：
- 按 a 排序
- a 相同，按 b 排序
- a、b 相同，按 c 排序

最左匹配原理：
- 索引有序的前提是前面的列确定
- WHERE a = 1：确定 a，b、c 有序
- WHERE a = 1 AND b = 2：确定 a、b，c 有序
- WHERE b = 2：a 不确定，b 无序，索引无效

范围查询后失效：
- WHERE a = 1 AND b > 2 AND c = 3
- a、b 可用索引，c 无序（b 不确定）

4. Index Condition Pushdown（ICP）

ICP：索引条件下推

没有 ICP：
- 存储引擎：通过索引找到满足索引条件的记录
- 返回给 Server 层
- Server 层：根据 WHERE 其他条件过滤

有 ICP：
- 存储引擎：索引条件 + 部分 WHERE 条件都在引擎层过滤
- 只返回满足全部条件的记录

减少：回表次数 + 返回记录数

三、MVCC 底层原理

1. 隐藏字段

每行记录有三个隐藏字段：

DB_TRX_ID（6字节）：
- 最后修改该行的事务 ID

DB_ROLL_PTR（7字节）：
- 回滚指针，指向 undo log

DB_ROW_ID（6字节）：
- 隐藏主键（无主键时使用）

2. Undo Log 结构

Undo Log 类型：

Insert Undo Log：
- 插入操作产生
- 只在事务回滚时需要
- 事务提交后可立即删除

Update Undo Log：
- 更新/删除操作产生
- 用于回滚和 MVCC
- 事务提交后不能立即删除

版本链：
- 最新数据指向最新 undo log
- 最新 undo log 指向上一版本 undo log
- 形成：数据 → undo1 → undo2 → undo3...

3. Read View 结构

Read View：
- 某次查询时的快照
- 决定哪些版本可见

字段：
m_ids：生成 Read View 时活跃的事务 ID 列表
min_trx_id：m_ids 中最小的事务 ID
max_trx_id：下一个要分配的事务 ID
creator_trx_id：生成 Read View 的事务 ID

可见性判断：
trx_id = DB_TRX_ID（数据的事务 ID）

1. trx_id == creator_trx_id
   → 可见（自己修改的）

2. trx_id < min_trx_id
   → 可见（Read View 生成前已提交）

3. trx_id >= max_trx_id
   → 不可见（Read View 生成后新事务）

4. min_trx_id <= trx_id < max_trx_id
   → 检查 trx_id 是否在 m_ids 中
   → 在：不可见（事务未提交）
   → 不在：可见（事务已提交）

4. 不同隔离级别的 Read View

READ COMMITTED：
- 每次查询生成新 Read View
- 同一事务两次查询可能看到不同数据

REPEATABLE READ：
- 第一次查询生成 Read View，后续复用
- 同一事务所有查询用同一 Read View

四、锁底层原理

1. 锁类型实现

行锁（Record Lock）：
- 锁单行记录
- 在索引记录上加锁

间隙锁（Gap Lock）：
- 锁一个范围，不锁记录本身
- 防止其他事务插入

临键锁（Next-Key Lock）：
- Record Lock + Gap Lock
- 锁记录 + 记录前面的间隙

插入意向锁：
- 插入时产生
- 等待间隙锁释放

意向锁（表级）：
- IS（意向共享）：准备加 S 锁
- IX（意向排他）：准备加 X 锁
- 用于快速判断表是否有行锁

2. 锁加锁规则

加锁规则（RR 隔离级别）：

1. 唯一索引等值查询
   - 记录存在：Record Lock
   - 记录不存在：Gap Lock

2. 唯一索引范围查询
   - 满足条件的记录：Next-Key Lock

3. 非唯一索引等值查询
   - 记录存在：Next-Key Lock + 后面 Gap Lock

4. 无索引
   - 所有记录 Next-Key Lock（全表锁）

RC 隔离级别：
- 只有 Record Lock（无间隙锁）

五、事务底层原理

1. Redo Log 结构

Redo Log：
- 保证事务持久性
- 记录物理修改（页级别的修改）

结构：
- 固定大小文件（ib_logfile0, ib_logfile1）
- 循环写入
- 写满时触发刷脏页

刷盘策略（innodb_flush_log_at_trx_commit）：
- 0：每秒刷盘（可能丢 1 秒数据）
- 1：每次提交刷盘（默认，最安全）
- 2：每次提交写 OS 缓存，每秒刷盘

2. Undo Log 作用

Undo Log：
- 事务回滚
- MVCC 版本链

回滚流程：
1. 从 Undo Log 恢旧值
2. 恢复索引记录
3. 删除 Insert Undo Log

3. 两阶段提交

两阶段提交：保证 Redo Log 和 Binlog 一致性

流程：
1. Prepare 阶段
   - 写 Redo Log，标记为 prepare 状态
   - 持有锁

2. Commit 阶段
   - 写 Binlog
   - Redo Log 改为 commit 状态
   - 释放锁

崩溃恢复：
- Redo Log prepare + Binlog 存在：提交事务
- Redo Log prepare + Binlog 不存在：回滚事务

六、Buffer Pool 底层

Buffer Pool：
- 缓存数据页、索引页
- 默认占内存 70-80%

结构：
- 页数组：缓存的数据页
- LRU 列表：页淘汰策略
- Flush 列表：脏页列表
- Free 列表：空闲页列表

LRU 实现：
- 分区 LRU
- Young 区域：热数据（5/8）
- Old 区域：冷数据（3/8）
- 新读入的页先放 Old 区域
- 再次访问才移到 Young 区域
- 防止全表扫描污染热数据

页读取：
1. 检查 Buffer Pool 是否有页
2. 有：直接访问
3. 无：从 Free 列表获取页（或淘汰 LRU 尾部）
4. 从磁盘读取数据到页
5. 加入 LRU 列表

页修改：
1. 修改 Buffer Pool 页
2. 标记为脏页（加入 Flush 列表）
3. 写 Redo Log
4. 后台线程刷脏页到磁盘
